Core Feature #1: Smart Task Management

What Site Officers Can Do

  • 📋 Create Tasks: Define name, duration, resources needed, prerequisites, and conflicts
  • 🔗 Map Dependencies: Specify which tasks must complete before others start
  • ⚠️ Flag Conflicts: Mark tasks that cannot run simultaneously (e.g., competing for the same excavator)
  • 📊 View Task Status: See tasks waiting, in-progress, completed, or ready for approval
  • ✏️ Update & Delete: Modify tasks or remove them if plans change
Example: Foundation task must complete before wall erection. System won't let walls start until foundation is approved.

Core Feature #2: Real-Time Resource Tracking

Three Resource Categories

  • 🔴 Rental Resources: Expensive equipment with daily charges (excavators, cranes). System highlights these to prevent idle costs.
  • ⏰ Expiring Resources: Materials with expiry dates (concrete, sealants). System warns before they become unusable.
  • 📦 Standard Resources: Regular materials and labor (rebar, lumber, workers)

What the System Does

  • Tracks total available quantity for each resource
  • Reserves resources when tasks start
  • Automatically consumes resources when tasks complete
  • Alerts when quantities run low

Core Feature #3: Intelligent Alert System

Real-Time Notifications

  • 🚨 Resource Shortage Alerts: System detects when current + upcoming task needs exceed available resources
  • ⏳ Expiry Warnings: "Cement expires in 3 days—use it now or lose it"
  • 💰 Rental Cost Alerts: "Excavator sitting idle—complete task by Friday or pay extra $500"
  • 🔗 Dependency Notifications: "Wall task ready to start—all prerequisites complete, resources available"
  • 📸 Approval Requests: HQ instantly notified when field supervisor uploads completion photos
Impact: Issues caught before they become expensive problems

System Architecture

Three-Tier Design

📱 Frontend (React): Mobile-first web interface running in user's browser. Connects to backend via REST API.
🖥️ Backend (Node.js + Express): Business logic, data processing, real-time events. Manages task dependencies, resource calculations, and approvals.
🗄️ Database (PostgreSQL): Stores all users, tasks, resources, projects, photos. Ensures data integrity with ACID compliance.
⚡ Real-Time (Socket.io): Instant notifications pushed to users when events occur (no refresh needed).

Technology Stack

Frontend

  • 🎨 React 18: Component-based UI, virtual DOM for performance
  • 📱 Mobile-First CSS: Responsive from 320px phones to 1920px desktops

Backend

  • 🚀 Node.js: Fast, non-blocking server
  • 🔌 Express: REST API framework
  • ⚡ Socket.io: Real-time bidirectional communication

Data & Security

  • 🗄️ PostgreSQL: Relational database for complex construction data
  • 🔐 Prisma ORM: Type-safe database queries
  • 🔒 Better-Auth: JWT-based authentication with hashed passwords

Data Model: Core Entities

Key Relationships

  • Users: Have a Role (Site Officer, HQ, Admin)
  • Projects: Assigned to one Site Officer, contain many Tasks
  • Tasks: Have Prerequisites (must complete first), Conflicts (can't run together), Required Resources
  • Resources: Have Types (rental, expiring, standard), allocated to Projects, consumed by Tasks
  • Photos: Evidence uploaded for task completion approval
Example: Foundation task requires 500 bags of cement. When task starts, 500 bags are reserved. When task completes, 500 bags are consumed and deducted from inventory.

Smart Algorithm #1: Task Readiness Detection

Can This Task Start?

The system checks THREE criteria before allowing a task to start:

  • ✅ Prerequisites Met: All dependent tasks are APPROVED
  • ✅ Resources Available: All required materials/labor are in stock
  • ✅ No Conflicts: No conflicting tasks are currently running
Example:
• Roof task needs 1000 shingles (only 800 available) → Can't start
• Roof task needs completion of wall (wall still in-progress) → Can't start
• Roof task + waterproofing both need crane (waterproofing running) → Can't start
• All conditions met → Task starts immediately!

Smart Algorithm #2: Concurrent Task Grouping

What Can Run in Parallel?

The system analyzes all waiting tasks and groups those that can run simultaneously:

  • 🔍 Identifies tasks with NO dependency relationships
  • 🔍 Verifies no resource conflicts between them
  • 🔍 Groups them into "batches" that can start together
Example:
• Task A (Interior painting) + Task B (Exterior landscaping) = Can run in parallel ✓
• Task C (Install plumbing) + Task D (Install electrical) = Can run in parallel ✓
• But all 4 together? No—painting needs electrical to be done first ✗
• System groups: [Paint + Landscape] then [Plumbing + Electrical] then [Paint + Landscape]

Smart Algorithm #3: Resource Shortage Prediction

Predict Problems Before They Happen

When a task completes, the system:

  • 1️⃣ Looks at the next waiting tasks
  • 2️⃣ Calculates total resources needed by next 2-3 tasks
  • 3️⃣ Compares to current available inventory
  • 4️⃣ If shortage detected → alerts HQ immediately
Example:
Current stock: 200 bricks
Next task needs: 500 bricks (shortage of 300)
Task after that needs: 400 bricks
Total shortage: 700 bricks
System instantly notifies HQ: "Need 700 bricks ASAP to keep workflow moving"
